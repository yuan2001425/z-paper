## 一、精准翻译

学术论文翻译的核心难题不是语言，而是**术语一致性**和**排版还原**。z-paper 通过六阶段并行流水线解决这两个问题。

```
上传 PDF
  → 重复检测（与已有论文标题相似度对比，避免重复收录）
  → MinerU 解析（识别多栏/公式/表格/图注）
  → 并行文本清理（8 并发）：合并 PDF 断行、修正 OCR 错误、规范 LaTeX 格式
  → 术语提取：自动识别领域专有名词
  → [暂停] 用户审查新术语，确认每个词的译法（仅翻译 / 保留原文 / 翻译并标注原文）
  → 带术语表并行翻译段落（8 并发，DeepSeek）+ 图片翻译（3 并发，Qwen）
  → 生成双栏译文
```

**关键设计：术语审查检查点**

大多数翻译工具在遇到新术语时会随机翻译，导致同一术语在论文不同位置出现不同译名。z-paper 在翻译前强制暂停，让用户确认每一个新术语的译法，确认后保存到个人词库，后续所有翻译自动复用，保证全文一致。

**双层词汇系统**

- **个人词库**：跨论文通用，自定义每个术语的处理策略
- **领域词库**：11 个预设学科（计算机科学、数学、物理学、生物学等）公用参考词表
- 每次翻译快照词库状态，译文可复现

**阅读体验**

- 中英双栏并排，支持切换单栏
- LaTeX 数学公式 KaTeX 实时渲染
- 图片 Lightbox 放大，原图与译图对比
- 文内引用 `[1]` 可点击跳转参考文献列表
- 精确到段落的划选批注系统

## 二、知识库对话

### 设计思想：工具优先，而非塞满 Context

目前大多数"论文问答"产品的做法是 RAG：把检索出的文本块塞进 context，让模型回答。这个方案有明显上限——检索质量决定回答质量，模型没有主动探索的能力。

z-paper 的设计思想借鉴自 **[Claude Code](https://claude.ai/code)**：**不预先决定给模型看什么，而是给 Agent 配备一套工具，让模型自主决定每一步该读哪里、读多少**。Context 是稀缺资源，Agent 的价值在于它知道如何高效使用它。

### 两个关键分离

**位置与内容分离**

工具返回的内容不需要永久驻留在 context 里。压缩时只保留索引（在哪里找到了什么），丢弃正文。正文随时可以通过工具重新获取——它是幂等的。这和人读论文的方式一样：你不会把整篇论文背下来，但你记得"第三节讲过这个，去翻一下"。

**检索与阅读分离**

搜索类工具负责定位，读取类工具负责精读。Agent 先定位、再按需深读，而不是一次性把所有内容塞进来。

### 14 个工具构成完整的导航与创作能力

| 工具 | 功能 |
| --- | --- |
| `search_papers` | 在全库元数据（标题/摘要/关键词）中搜索 |
| `get_paper_outline` | 获取论文章节大纲 |
| `get_paper_metadata` | 快速获取论文摘要/作者/关键词 |
| `search_in_paper` | 在单篇论文全文中关键词检索 |
| `search_across_papers` | **跨全库**全文搜索，一次扫遍所有论文 |
| `get_paper_section` | 获取指定章节**完整文本**（无段落数量限制） |
| `get_paragraph_context` | 获取某段落的前后文 |
| `get_references` | 获取参考文献列表 |
| `get_annotations` | 获取某篇论文的所有个人批注 |
| `search_annotations` | **跨全库**搜索个人批注（中文分词 + 拼音匹配） |
| `search_chat_history` | 搜索历史对话记录 |
| `query_database` | 执行自定义 SQL 查询（最后手段，处理其他工具无法表达的复杂查询） |
| `generate_image` | 根据描述生成图片（文生图，可视化论文概念/流程/对比） |
| `edit_image` | 对已有图片按指令修改 |

Agent 每一步自主决定调哪个工具、调几次，直到认为信息足够才输出回答。没有固定流程，调用策略完全由模型根据问题推理。

### AI 图片生成

对话时可直接要求 Agent 生成图表，例如"用图示说明 Transformer 的注意力机制"。Agent 会调用 `generate_image` 工具，将生成的图片内嵌在回答中。图片支持：

- 点击放大，鼠标滚轮缩放（0.5× ~ 5×），放大后可拖拽平移
- 一键下载
- 点击"编辑图片"后输入修改指令，Agent 调用 `edit_image` 对原图进行调整

### 两层记忆架构

- **L1 工作记忆**：当前对话上下文，每轮自动注入论文目录和个人词表
- **L2 冷数据**：全部历史对话持久化在本地数据库，Agent 通过 `search_chat_history` 工具按需检索。每个会话完全独立，不自动注入历史，避免错误结论跨会话污染

### 五级上下文压缩，支持超长研究会话

| 级别 | 触发条件 | 操作 | API 开销 |
| --- | --- | --- | --- |
| L1 剪裁 | >4 轮 | 旧工具结果压缩为结构索引（保留"在哪找到"，丢弃正文） | 无 |
| L2 微压缩 | >8 轮 | 旧工具调用折叠为单行摘要 | 无 |
| L3 折叠 | >12 轮 | 旧对话内容截短 | 无 |
| L4 自动压缩 | >16 轮 | LLM 生成全量摘要替换旧轮次 | 有 |
| L5 应急压缩 | 超 80K 字符 | 强制保留最近 3 轮 | 无 |

熔断器：L4 连续失败 3 次后停止重试，不浪费 API 调用。

### 其他细节

- **双语搜索**：中文提问时自动同时搜索中英文关键词，中文人名还会尝试拼音，最大化召回率
- **流式输出**：工具调用阶段非流式保证 JSON 格式准确，最终回答 SSE 流式推送，实时展示每步工具调用过程
- **引用溯源**：每条回答附带引用卡片，点击后打开原文并**自动滚动定位到对应段落**（精确到 block 级别）
- **会话 Minimap**：对话页面右侧缩略导航栏，实时追踪当前位置

## 快速启动（源码）

**系统要求：** Python 3.11+、Node.js 18+

```bash
git clone https://github.com/yuan2001425/z-paper.git
cd z-paper

# Windows
start.bat

# Linux / macOS
bash start.sh
```

首次启动跳转配置页，填写三个 API Key：

| Key | 用途 | 申请 |
| --- | --- | --- |
| DeepSeek API Key | 翻译 + 对话 | https://platform.deepseek.com |
| 通义千问 API Key | 图片识别 + 文本处理 + 图片生成 | https://dashscope.console.aliyun.com |
| MinerU API Key | PDF 解析 | https://mineru.net |
